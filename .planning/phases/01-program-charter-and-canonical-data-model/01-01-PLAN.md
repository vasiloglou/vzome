# Phase 1 Plan 01: Lock The Ingestion Contract

## Objective

- **What:** Define the contract, source/tooling inventory, registry, and
  integration boundaries for the Material Design Data Ingestion project before
  building Tier 1 adapters.
- **Why:** The current repo has a strong but narrow ingest path. We need a
  canonical multi-source ingestion design that preserves provenance, reflects
  the real source ecosystem, and fits the existing `materials-discovery`
  architecture.
- **Output:** Three phase design artifacts:
  `01-DATA-CONTRACT.md`, `01-SOURCE-REGISTRY.md`, and
  `01-INTEGRATION-DESIGN.md`.

## Verification

Verify the phase output by checking four things:

1. The three design artifacts cover all Phase 1 deliverables in `.planning/ROADMAP.md`.
2. The contract and registry decisions explicitly satisfy `DATA-01`, `DATA-02`,
   `OPS-01`, and `OPS-02` in `.planning/REQUIREMENTS.md`.
3. The proposed design is consistent with the existing architecture documented
   in `materials-discovery/developers-docs/` and with the current code seams in
   `cli.py`, `common/schema.py`, `data/ingest_hypodx.py`, and
   `backends/registry.py`.
4. The integration design leaves a clean path to Phase 2 adapter implementation
   and Phase 3 `mdisc ingest` integration.

## Success Criteria

- [ ] The phase outputs define a canonical raw-source record that is distinct
  from the current processed `IngestRecord`.
- [ ] The source registry makes the Tier 1 adapter wave and access/licensing
  posture explicit while preserving a broader watchlist inventory.
- [ ] The integration design preserves the current operator-facing CLI and
  file-backed architecture.
- [ ] The resulting artifacts are concrete enough to start Phase 2 without
  reopening basic contract questions.
