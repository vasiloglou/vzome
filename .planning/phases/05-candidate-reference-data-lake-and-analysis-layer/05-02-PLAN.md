Create `materials-discovery/src/materials_discovery/lake/compare.py`:

Define Pydantic model `MetricDistribution`:
  - `mean: float`
  - `min: float`
  - `max: float`
  - `std: float`
  - `count: int`

Define Pydantic model `LaneSnapshot`:
  - `system: str`
  - `lane_id: str`
  - `backend_mode: str`
  - `source_keys: list[str]`
  - `reference_pack_id: str | None`
  - `gate_results: dict[str, bool]` (from release_gate)
  - `summary_metrics: dict[str, float]` (from report summary)
  - `metric_distributions: dict[str, MetricDistribution]` (computed from report entries)
  - `pack_path: str` (workspace-relative)
  - `report_path: str | None` (workspace-relative, resolved from stage_manifest_paths)

Implement class method `LaneSnapshot.from_benchmark_pack(pack_path: Path, workspace: Path) -> LaneSnapshot`:
  - Load benchmark_pack.json
  - Extract benchmark_context for lane identity fields
  - Extract report_metrics.release_gate and report_metrics.summary
  - Dereference stage_manifest_paths["report"] to find the report JSON
  - If report JSON exists, load it and compute MetricDistribution for each metric across entries:
    - hifi_score, stability_probability, ood_score, xrd_confidence, xrd_distinctiveness, delta_e_proxy_hull_ev_per_atom (from evidence), uncertainty_ev_per_atom (from evidence), md_stability_score (from evidence)
  - Use statistics.mean, min, max, statistics.stdev (with fallback to 0.0 for single-element lists)
  - If report JSON missing, fall back to report_metrics from the pack itself (summary metrics only, no distributions). Log warning.
  - All paths stored as workspace-relative using workspace_relative()

This lane-centric internal model addresses review concern #6: system-vs-system is just a preset view where both lanes happen to be different systems. The same model supports future source-vs-source or backend-vs-backend comparisons.

Define Pydantic model `GateDelta`:
  - `gate_name: str`
  - `lane_a: bool`
  - `lane_b: bool`
  - `status: str` (one of "both_pass", "both_fail", "regression", "improvement")

Define Pydantic model `MetricDelta`:
  - `metric_name: str`
  - `lane_a: MetricDistribution | None`
  - `lane_b: MetricDistribution | None`
  - `delta_mean: float | None`

Define Pydantic model `ComparisonResult`:
  - `schema_version: str` = "comparison/v1"
  - `generated_at_utc: str`
  - `lane_a: LaneSnapshot`
  - `lane_b: LaneSnapshot`
  - `gate_deltas: list[GateDelta]`
  - `metric_deltas: list[MetricDelta]`

Implement `compare_benchmark_packs(pack_a: Path, pack_b: Path, workspace: Path | None = None) -> ComparisonResult`:
  - Build LaneSnapshot for each
  - Compute gate deltas for each gate in release_gate
  - Compute metric deltas: for each metric that has distributions in both lanes, compute delta_mean = lane_b.mean - lane_a.mean
  - Return ComparisonResult

Implement `write_comparison(result: ComparisonResult, output_dir: Path | None = None) -> Path`:
  - Default output_dir: workspace_root() / "data" / "comparisons"
  - Filename: `{lane_a.system}_vs_{lane_b.system}.json` (slugified)
  - Write JSON, return path. Per D-06.

Implement `format_comparison_table(result: ComparisonResult) -> str`:
  - Multi-line formatted table suitable for terminal output
  - Header: "Lane A: {system} ({lane_id}) vs Lane B: {system} ({lane_id})"
  - Gate section: gate_name | lane_a | lane_b | status
  - Metric section: metric_name | mean_a | mean_b | delta | std_a | std_b
  - Per D-06 dual-format requirement.

Create `materials-discovery/tests/test_lake_compare.py`:
