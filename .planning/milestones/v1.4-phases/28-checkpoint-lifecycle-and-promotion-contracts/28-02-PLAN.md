Extend materials-discovery/src/materials_discovery/llm/checkpoints.py with
family lifecycle helpers.

Add typed helpers to:
- load or lazily create the lifecycle index for a checkpoint family
- enumerate registered checkpoints by family
- auto-enroll a newly registered checkpoint as a candidate when its
  registration carries checkpoint_family
- apply promotion actions
- apply retirement actions
- resolve the active promoted member for later Phase 29 integration

Use an explicit stale-write guard based on an expected integer revision from
the family lifecycle index, and optionally an expected promoted checkpoint id.
On every successful lifecycle mutation, increment the family revision by 1.

Fail with clear operator-facing errors when:
- the requested checkpoint is unknown to the family
- multiple promoted members would result
- a stale revision is supplied
- the caller supplied checkpoint_family plus checkpoint_id that do not match
  the registered family membership
- a retirement action would leave the family in an ambiguous state

Make the enrollment rule explicit: registering a checkpoint with
checkpoint_family must ensure the lifecycle index exists and contains that
checkpoint as a candidate member, while legacy registrations without
checkpoint_family remain untouched.

Make the action contract explicit:
- promotion specs accept repo-relative evidence file paths, not free-text
  evidence notes
- action artifacts use the revision-based paths from Plan 01
- demotion means promoting a different checkpoint; do not add a separate
  llm-demote-checkpoint command in this phase

Do not yet wire these helpers into generation, launch, or replay. Their job in
Phase 28 is to establish the authoritative family lifecycle contract.

Update exports in materials-discovery/src/materials_discovery/llm/__init__.py
for any new helper surfaces.

Extend materials-discovery/tests/test_llm_checkpoint_registry.py with the
family-lifecycle and stale/conflict cases above.
