Add a new command to `materials-discovery/src/materials_discovery/cli.py`:
`mdisc llm-launch --campaign-spec PATH [--out PATH]`

Behavior:
- load and validate `LlmCampaignSpec`
- load the system config from `spec.launch_baseline.system_config_path`
- require the current config hash to equal `spec.launch_baseline.system_config_hash`; otherwise raise `ValueError("campaign spec config hash does not match current config: <path> changed since approval; re-approval may be required")`
- create a new launch ID and campaign launch directory
- print the `launch_id` as soon as it is created so operators can correlate later failures with on-disk launch artifacts
- write `resolved_launch.json` before provider execution
- call `resolve_campaign_launch(...)` to get the copied config and resolved overlay
- call `generate_llm_candidates(...)` with:
  - the copied config
  - `count=spec.launch_baseline.default_count`
  - the resolved `--out` path or the standard default candidates path
  - additive `prompt_instruction_deltas`
  - additive `campaign_metadata`
- build or update `launch_summary.json` on both success and failure
- on success, print the typed launch summary JSON
- on failure, write a failed launch summary and exit with code 2

`resolved_launch.json` must also record:
- whether `--out` overrode the standard candidates output path
- the final effective candidates path used for the launch
- whether the resolved model lane came from a configured-lane match or the baseline fallback

Concrete launch summary fields to populate:
- `launch_id`
- `campaign_id`
- `campaign_spec_path`
- `proposal_id`
- `approval_id`
- `status`
- `resolved_launch_path`
- `run_manifest_path`
- `llm_generate_manifest_path`
- `candidates_path`
- `error_kind`
- `error_message`

CLI error handling must follow repo conventions:
- catch `FileNotFoundError`, `ValidationError`, `ValueError`, and `RuntimeError`
- emit `llm-launch failed: ...`
- exit 2

Config-drift failures must stay especially operator-friendly:
- mention the system-config path from `spec.launch_baseline.system_config_path`
- mention both the pinned and current hash when available
- mention that the spec should be re-approved after intentional config changes
